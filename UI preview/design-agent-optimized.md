# 华彩经营管理平台 UI 设计规范

版本：2026 浅蓝企业 SaaS 风格优化版  
适用项目：华彩经营管理平台前端  
技术栈：Vue 3 + TypeScript + Vite + Element Plus  
参考文件：

- `UI preview/login.html`
- `UI preview/dashboard.html`
- `UI preview/design agent.md`
- `frontend/src/style.css`
- `frontend/src/layout/AppLayout.vue`

---

## 1. 规范用途

本规范用于指导 design agent、前端 agent 和工程师，在不改变核心业务信息架构、接口逻辑、权限逻辑和业务文案的前提下，将华彩经营管理平台统一升级为更现代、更精致、更适合长期使用的企业级后台界面。

它不是静态 HTML 交付规范，而是当前 Vue + Element Plus 项目的 UI 落地规范。所有改造应优先复用现有组件、路由、store、API、Element Plus 组件和全局样式体系。

---

## 2. 设计目标

整体目标是做一个高完成度、可落地、适合真实业务系统长期使用的 B 端经营管理平台，而不是营销官网、展示页或炫技型概念稿。

关键词：

- 浅蓝白
- 企业级
- 科技感
- 通透
- 克制
- 精致
- 高级感
- 清晰层级
- 轻玻璃感
- 数据化
- 长期可维护
- B 端专业感

设计判断标准：

- 第一眼像高端企业 SaaS，而不是旧式后台系统。
- 页面明亮、轻盈、可信赖。
- 信息密度适合业务操作，不因为追求视觉而降低效率。
- 登录页、工作台、列表页、详情页、表单页使用同一套设计语言。

## 3. 总体视觉原则

### 3.1 风格定位

华彩系统是经营管理、客户管理、借贷管理、财务管理、人事管理、系统管理和审批流的综合后台。视觉应体现稳定、专业、可信赖，不能走消费级、游戏化、赛博化或营销化方向。

应具备：

- 明亮但不刺眼
- 精致但不花哨
- 有科技感但不未来化过度
- 有品牌感但不影响业务阅读
- 有留白但不牺牲 B 端效率

### 3.2 优先级

UI 改造优先级如下：

1. 保留现有业务结构、路由、权限和接口逻辑。
2. 建立统一 token、Element Plus 基础覆盖和布局体系。
3. 优先改造登录页、工作台和高频业务列表页。
4. 将可复用页面样式沉淀到全局样式或通用组件。
5. 再逐步覆盖低频页面和边缘状态。

### 3.3 不做什么

- 不做纯静态 HTML 交付。
- 不绕开 Vue 组件体系。
- 不引入新的 UI 框架替代 Element Plus。
- 不把后台改成官网落地页。
- 不为了装饰删除、隐藏或弱化关键业务信息。
- 不随意改业务文案、字段、权限、接口和数据流。

## 4. 色彩系统

### 4.1 品牌主色

建议在 `frontend/src/style.css` 中建立或更新这些 token：

```css
:root {
  --hc-primary: #4A85FF;
  --hc-primary-deep: #3167F3;
  --hc-primary-soft: rgba(74, 133, 255, 0.12);
  --hc-primary-light: #8BB6FF;
}
```

使用规则：

- 主按钮、重点操作、当前导航、链接操作使用品牌蓝。
- 大面积背景不要使用高饱和蓝色。
- 品牌蓝应主要作为强调色，而不是整页铺满。

### 4.2 页面背景

```css
:root {
  --hc-bg: #EEF4FF;
  --hc-bg-soft: #F8FBFF;
  --hc-surface: #FFFFFF;
  --hc-surface-glass: rgba(255, 255, 255, 0.78);
}
```

推荐背景：

- 浅蓝白纵向渐变。
- 少量低透明 radial glow。
- 登录页、工作台、侧栏可以有轻微玻璃感。
- 业务列表页背景保持干净，避免干扰表格阅读。

### 4.3 文字颜色

```css
:root {
  --hc-text: #122D5F;
  --hc-text-soft: #7085AD;
  --hc-text-muted: #95A7C4;
}
```

使用规则：

- 标题、表格主字段使用 `--hc-text`。
- 描述、辅助信息、组织名、提示文字使用 `--hc-text-soft`。
- 占位、弱提示、元信息使用 `--hc-text-muted`。
- 不使用纯黑作为主文字。

### 4.4 边框和分割线

```css
:root {
  --hc-border: #DBE8FF;
  --hc-border-strong: #CDDFFF;
  --hc-divider: #EDF3FF;
}
```

边框必须轻，不使用黑灰粗边，不使用高对比硬描边。表格、筛选条、卡片、用户卡片、弹层都应保持浅蓝边框体系。

### 4.5 功能色

```css
:root {
  --hc-success: #18A69D;
  --hc-warning: #DB7D14;
  --hc-danger: #EF7386;
}
```

功能色只用于状态、优先级、风险提示和少量图标。不要大面积铺满，不要使用荧光色。

## 5. 圆角、阴影、边框

### 5.1 圆角体系

```css
:root {
  --hc-radius-sm: 14px;
  --hc-radius-md: 18px;
  --hc-radius-card: 24px;
  --hc-radius-panel: 32px;
}
```

建议映射：

- 小图标底板、Tag：14px-16px
- 输入框、按钮、筛选控件：16px-18px
- 普通卡片：22px-26px
- AppLayout 侧栏、登录页主面板、工作台大面板：30px-32px

注意：圆角可以更大，但不能让表格、表单控件变得幼稚或玩具化。

### 5.2 阴影体系

```css
:root {
  --hc-shadow-card: 0 10px 26px rgba(88, 123, 191, 0.08);
  --hc-shadow-panel: 0 18px 50px rgba(77, 120, 200, 0.12);
  --hc-shadow-primary: 0 18px 38px rgba(49, 103, 243, 0.25);
}
```

使用规则：

- 卡片阴影浅、柔、低对比。
- 主按钮允许有蓝色投影，但面积和透明度要克制。
- 表格内部不要使用重阴影。
- 弹层、抽屉、下拉菜单可以比卡片稍强，但不能发脏。

## 6. 布局规范

### 6.1 应用壳层

`frontend/src/layout/AppLayout.vue` 是登录后所有页面的视觉底座，应优先改造。

结构保持：

- 左侧导航栏
- 右侧内容区
- 顶部标题和用户操作区
- 主内容 `router-view`

视觉要求：

- 整体外层使用浅蓝白渐变背景。
- 左侧侧栏使用白色半透明、浅蓝边框、柔和阴影和轻玻璃感。
- 侧栏圆角建议 30px-32px。
- 导航图标使用统一浅蓝白底板。
- 当前导航项使用浅蓝底或浅蓝渐变，不使用深色硬块。
- 右侧内容区留白更舒展，避免页面贴边。

### 6.2 页面内容

通用页面使用 `.page-shell`、`.card`、`.card__section`、`.page-heading` 等全局样式承载视觉一致性。

建议：

- 页面模块间距：18px-24px。
- 卡片内边距：20px-24px。
- 大页面顶部介绍区可以更宽松。
- 高密度表格页允许相对紧凑，但仍需保持行高和可读性。

### 6.3 响应式

桌面端优先，但必须保证基础响应式：

- 1180px 以下：顶部操作和页面区块允许换行。
- 960px 以下：侧栏可堆叠到顶部，保留导航可用性。
- 720px 以下：按钮、用户信息、标题不能互相遮挡或溢出。
- 表格允许横向滚动，但页面整体不能出现无意义的横向溢出。

## 7. Element Plus 基础覆盖规范

本项目应继续使用 Element Plus。UI 改造重点是通过 CSS token 和全局覆盖提升质感，而不是替换组件库。

### 7.1 Button

主按钮：

- 蓝色纵向渐变。
- 白色文字。
- 圆角 16px-18px。
- 柔和蓝色阴影。
- hover 稍微提亮或上浮，不要大幅跳变。

次按钮：

- 白底或浅蓝白底。
- 浅蓝边框。
- 深蓝灰文字。
- hover 时轻微蓝底。

文字按钮：

- 用于表格操作列和轻量操作。
- 保持清晰蓝色，不要加过重背景。

### 7.2 Input / Select / DatePicker

- 高度应比传统后台更舒展。
- 圆角 16px-18px。
- 白底 + 浅蓝边框。
- focus 态使用浅蓝外发光或边框加深。
- placeholder 使用弱化文字色。

### 7.3 Table

表格是 B 端效率核心，不能为了视觉牺牲密度和扫描效率。

要求：

- 表头浅蓝白背景。
- 表头文字深蓝灰或次级文字色。
- 行高舒展但不过度放大。
- 分割线使用 `--hc-divider`。
- hover 态轻微浅蓝。
- 固定列阴影要轻。
- 操作列链接使用品牌蓝。

禁止：

- 深灰表头。
- 重横线。
- 单元格过度留白。
- 表格内大量卡片化。

### 7.4 Tag

- 胶囊形态。
- 柔和底色。
- 状态色不使用纯色硬块。
- 高风险/驳回：浅粉底 + 红字。
- 中提醒/待处理：浅橙底 + 橙字。
- 成功/已完成：浅青底 + 绿字。
- 普通/未开始：浅蓝灰底 + 蓝灰文字。

### 7.5 Dialog / Drawer / Dropdown / Pagination

- 弹层圆角统一。
- 边框和阴影沿用浅蓝体系。
- Dropdown 菜单项 hover 使用浅蓝底。
- Pagination 当前页使用品牌蓝，但不要过重。

## 8. 组件和页面模式

### 8.1 Card

卡片是系统核心视觉单位。

要求：

- 白色或浅蓝白背景。
- 1px 浅蓝边框。
- 22px-26px 圆角。
- 柔和阴影。
- 可有极轻微渐变和高光。

不要：

- 灰底传统后台卡片。
- 重阴影。
- 强烈拟物纹理。
- 卡片套卡片造成层级混乱。

### 8.2 Page Intro

用于客户、借贷、财务、系统等业务页面顶部说明区。

建议结构：

- eyebrow：模块路径或业务域。
- h2：页面主标题。
- p：简短业务说明。
- actions：主操作、次操作、更多操作。

视觉：

- 标题清晰。
- 描述不超过两行。
- 操作区与标题区对齐。
- 移动端允许上下堆叠。

### 8.3 Filter Toolbar

筛选区应清晰、轻量、容易操作。

- 使用浅蓝白底或白底。
- 边框轻。
- 输入框宽度稳定。
- 查询/重置按钮固定在右侧或底部。
- 不要把筛选区做成复杂装饰卡片。

### 8.4 Detail / Form Page

详情和表单页应注重清晰分组：

- 基础信息、合同信息、风险信息、负债信息等用清晰区块。
- 分组标题保持一致。
- 表单控件对齐。
- 错误提示靠近对应字段。
- 保存、提交、返回等操作固定在明显位置。

## 9. 图标规范

当前项目使用 `@element-plus/icons-vue`，应优先继续复用，避免混入随机第三方图标。

要求：

- 图标风格统一。
- 导航图标放在浅蓝白小底板里。
- 功能按钮图标只在能增强识别时使用。
- 表格操作不强制加图标，避免噪音。
- 不使用 emoji 作为业务图标。

允许：

- 登录页、工作台指标卡、侧栏底部使用 CSS/SVG 装饰图形。
- 装饰图形必须低对比、浅蓝体系、不抢内容。

## 10. 页面级规范

### 10.1 登录页

对应文件：`frontend/src/views/auth/LoginView.vue`  
参考文件：`UI preview/login.html`

目标：

- 左侧品牌展示区。
- 右侧登录面板。
- 浅蓝白、玻璃感、科技感。
- 保留现有登录逻辑、表单校验和错误提示。

必须移除：

- 绿橙渐变。
- 廉价后台登录页气质。
- 与主系统不一致的配色。

### 10.2 工作台

对应文件：`frontend/src/views/dashboard/WorkbenchView.vue`  
参考文件：`UI preview/dashboard.html`

目标：

- 顶部操作轻量化。
- 指标卡更有层次。
- 最近业务记录表格更通透。
- 今日待办、运行提醒、快捷入口保持右侧辅助区。
- 保留当前接口和数据结构。

不要：

- 重写 API。
- 改变数据字段含义。
- 删除业务模块。

### 10.3 业务列表页

代表页面：

- `frontend/src/views/customer/CustomerListView.vue`
- `frontend/src/views/loan/LoanOrderListView.vue`
- `frontend/src/views/finance/*ListView.vue`
- `frontend/src/views/workflow/*View.vue`

目标：

- 统一 page intro。
- 统一筛选条。
- 统一表格样式。
- 统一分页和操作列。
- 保持字段完整和扫描效率。

### 10.4 系统管理页

系统管理页面通常信息层级复杂，应优先保证：

- 权限、角色、组织、用户关系清楚。
- 树、表格、表单之间间距稳定。
- 不要过度装饰权限配置区域。

## 11. 代码实现规范

### 11.1 技术边界

本项目不是静态 HTML 项目。Agent 应在现有 Vue 工程内修改：

- `.vue` 单文件组件
- `frontend/src/style.css`
- 必要时新增轻量通用组件或样式文件

不得把 `UI preview/*.html` 直接复制为生产页面。

### 11.2 修改原则

- 保留业务逻辑。
- 保留现有 API 调用。
- 保留 router、store、权限判断。
- 优先改样式和模板结构。
- 抽象只在能减少真实重复时添加。
- 不引入新的 UI 框架。
- 不做无关重构。

### 11.3 CSS 组织

优先级：

1. 全局 token：`frontend/src/style.css`
2. Element Plus 基础覆盖：`frontend/src/style.css`
3. 通用页面 class：`frontend/src/style.css`
4. 页面特有样式：对应 `.vue` 的 scoped style

如果多个业务页面重复出现同一模式，应沉淀为全局 class，而不是复制粘贴多份 scoped CSS。

### 11.4 图形和装饰

优先使用：

- CSS gradient
- CSS radial glow
- 内联 SVG
- Element Plus icons

避免：

- 大量位图依赖。
- 随机素材。
- 复杂动画。
- 影响性能的滤镜叠加。

## 12. Agent 执行规则

执行 UI 改造时必须遵守：

1. 先读 `UI preview/design-agent-optimized.md`。
2. 再读对应参考 HTML 和当前 Vue 文件。
3. 保留现有业务结构、接口、权限和主要文案。
4. 优先统一 token、组件基础样式、页面模式。
5. 任何页面改造都要和 AppLayout、登录页、工作台保持同一设计语言。
6. 不机械换色，要检查间距、层级、圆角、阴影、表格密度和响应式。
7. 不引入大面积深色、紫色、绿橙、荧光色。
8. 不使用 emoji 图标。
9. 不删除业务模块。
10. 改完必须运行构建和相关测试。

推荐验证命令：

```bash
cd frontend && npm run build
cd frontend && npm run test:unit
```

如果失败，应判断是否由本次 UI 修改导致；如果是，应修复后再交付。

## 14. 质量验收清单

视觉：

- 是否符合浅蓝企业 SaaS 风格。
- 是否和登录页、工作台、业务页保持一致。
- 是否避免旧式 Bootstrap/传统后台气质。
- 阴影是否柔和。
- 边框是否轻。
- 圆角是否统一。

体验：

- 表格是否仍然高效可读。
- 表单是否易填写。
- 操作按钮主次是否清楚。
- 状态 Tag 是否可快速识别。
- 页面是否没有无意义横向溢出。

工程：

- 是否保留业务逻辑。
- 是否没有引入多余依赖。
- 是否没有复制静态 HTML 到 Vue 页面。
- 是否通过 `npm run build`。
- 是否通过相关测试。

## 15. 给 Agent 的简短提示词

请基于本规范，在现有 Vue 3 + Element Plus 项目内执行 UI 改造。使用统一的浅蓝企业 SaaS 设计语言，保留原有业务信息架构、接口逻辑、权限逻辑和主要文案。重点提升全局 token、Element Plus 基础质感、AppLayout、登录页、工作台、卡片、表格、筛选区、Tag、按钮、输入框、间距、阴影、边框和响应式表现。不要机械换色，不要输出静态 HTML，不要引入新 UI 框架，不要删除业务模块。交付前运行构建和测试。
