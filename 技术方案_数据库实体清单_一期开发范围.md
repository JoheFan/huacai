# 华彩系统技术方案、数据库实体清单与一期开发范围

## 1. 目标与前提

### 1.1 目标

基于当前原型，输出一版可以直接进入研发准备阶段的方案，覆盖：

- 技术架构推荐
- 数据库核心实体清单
- 一期开发范围与排除范围

### 1.2 当前判断

根据现有原型，系统本质上是一个“综合经营后台”，包含：

- 基础平台能力：登录、权限、组织、角色、日志
- 客户经营：客户档案、风险评估、负债、合同、商机、跟进
- 经营台账：我方借贷、银行借贷、还款、导入导出
- 财务流水：收入、支出、收款方
- 人事/OA：员工档案、工资、审批
- 分析：员工绩效、增量分析

### 1.3 方案前提

本方案按以下前提设计：

- 技术栈固定为 `Vue3 + Spring + MySQL`
- 一期优先做“能支撑核心业务跑起来”的范围
- 先不做微服务
- 先不做复杂工作流引擎
- 先不做重 BI 报表


## 2. 技术架构方案

### 2.1 可选方案

#### 方案 A：前后端分离的模块化单体应用

说明：

- 前端单独一个 Vue3 管理后台
- 后端一个 Spring Boot 应用
- 数据库一个 MySQL 实例
- 后端按业务域拆包，但仍为单体部署

优点：

- 开发效率最高
- 架构复杂度低
- 容易对齐当前原型的中后台结构
- 一期最适合快速交付

缺点：

- 后期业务继续膨胀时，需要做模块拆分治理

#### 方案 B：前后端分离的“大单体 CRUD 系统”

说明：

- 本质也是单体，但不做明确业务域边界
- 以页面和表为中心快速开发

优点：

- 初期开发最快

缺点：

- 后期维护成本高
- 借贷、客户、财务、人事会强耦合
- 原型虽多，但业务规则并不简单，这种方式后期返工概率高

#### 方案 C：微服务架构

说明：

- 按系统管理、客户、借贷、财务、人事、分析拆成多个服务

优点：

- 理论上扩展性最好

缺点：

- 当前阶段明显过度设计
- 团队成本、部署成本、联调成本高
- 原型规则尚未完全收敛，不适合现在拆服务

### 2.2 推荐方案

推荐采用：

- **方案 A：前后端分离的模块化单体应用**

推荐理由：

- 当前项目最大问题不是并发或海量流量，而是业务边界多、规则未完全固化
- 模块化单体最适合先把客户、借贷、财务这几条主线打通
- 后续如果人事、审批、分析模块持续膨胀，再按业务域拆服务也不晚


## 3. 推荐技术栈

### 3.1 前端

- `Vue 3`
- `TypeScript`
- `Vite`
- `Vue Router`
- `Pinia`
- `Element Plus`
- `Axios`
- `ECharts`
- `xlsx` 或浏览器侧导出组件

推荐原因：

- Vue 官方对 Vue 3 的推荐链路已经明确转向 `Vite + Vue Router + Pinia`
- Element Plus 适合当前原型这类典型后台表单、表格、弹窗、审批流页面

### 3.2 后端

- `Spring Boot 3.x`
- `Spring Web MVC`
- `Spring Security`
- `Spring Validation`
- `MyBatis-Plus`
- `OpenAPI / springdoc-openapi`
- `Apache POI`

推荐原因：

- Spring Boot 3 + Spring Security 足够支撑标准中后台认证鉴权
- MyBatis-Plus 适合当前大量列表页、表单页、单表/中等复杂查询的开发节奏
- Apache POI 适合实现 Excel 导入导出

### 3.3 数据库与存储

- `MySQL 8.0`
- 文件存储一期采用：
- 本地文件存储
- 数据库存储文件元数据

后续可扩展：

- MinIO
- 阿里云 OSS
- 腾讯云 COS

### 3.4 部署方式

- 前端：Nginx 托管静态资源
- 后端：Spring Boot Jar 部署
- 数据库：MySQL 单实例

### 3.5 一期不强依赖的组件

- Redis：可不在一期强制引入
- 消息队列：暂不引入
- 工作流引擎：暂不引入
- ES：暂不引入

- 一个组织下可以有多个用户
- 一个用户可以绑定多个角色
- 一个角色可以绑定多个菜单权限
- 一个客户可以有多条评估记录
- 一个客户可以有多条状态变更记录
- 一个客户可以有多条负债记录
- 一个客户可以有多份合同
- 一个客户可以有多条交易记录
- 一个客户可以有多条商机
- 一个商机可以有多条跟进记录
- 一个客户可以有多笔借贷记录
- 一笔借贷记录可以有多条还款明细
- 收入/支出都可以绑定附件

## 10. 当前最需要业务确认的点

- 客户主键到底是“客户ID”、统一社会信用代码，还是二者联合唯一
- “测试额度”“流量”“龙信商/第三方评分”的真实业务含义
- 借贷和还款的计算口径是否需要自动汇总
- “我方借贷”和“银行借贷”是否允许统一建模
- 财务收入/支出类型是否需要字典配置
- 增量模块的业务定义是什么，是否真的是一期刚需
- 审批流二期是否需要引入 BPM/workflow 引擎


## 11. 结论

结论如下：

- 当前项目适合采用 **Vue3 + Spring Boot 3 + MySQL 8 的前后端分离模块化单体方案**
- 一期建议先做 **基础平台 + 客户管理 + 商机跟进 + 借贷还款 + 财务基础**
- 人事审批、增量分析、绩效分析建议放入二期
- 数据库设计上建议优先围绕 **用户、客户、商机、借贷、还款、财务** 六大核心域展开
