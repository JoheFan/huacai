# 华彩客户管理重构设计

## 背景

当前一期客户模块已经完成了“客户列表 + 简化新增抽屉 + 真实接口可用”的最小闭环，但这套实现和最新确认的原型口径差距较大：

- 客户新增仍是简化抽屉，不是完整客户档案页
- 风险评估、负债登记、合同管理仍是客户子表骨架，没有独立页面形态
- 左侧菜单当前没有“客户管理”一级分组下的二级展开结构
- 风险评估、负债登记接口仍是占位返回

本次重构目标不是推翻系统基础设施，而是在保留现有登录、权限、菜单框架、真实 MySQL 和客户主数据链路的前提下，把客户经营模块收成“可交付的原型对齐版”。

## 目标

本轮完成后，客户经营相关模块应满足以下目标：

1. 左侧菜单调整为：
   - 一级分组：`客户管理`
   - 二级菜单：`客户管理 / 风险评估 / 负债登记`
   - `商机管理` 从原分组迁移到 `经营管理`
2. `客户管理` 页面支持打开完整客户档案页，而不是继续沿用现有简化抽屉
3. 客户档案页一次提交整张档案页，统一保存：
   - 基本信息
   - 风险评估记录
   - 负债登记记录
   - 合同记录
4. `风险评估` 页面与 `负债登记` 页面作为左侧独立二级入口存在，但数据源仍然来自客户详情子表
5. 风险评估与负债登记列表一行分别代表一条记录，并都支持客户筛选
6. 导入导出按钮本轮只保留视觉入口，置灰，不接真实能力

## 非目标

本轮不做以下事项：

- 不重做整体 UI 风格
- 不扩审批流、报表、复杂权限、导入导出真实能力
- 不把合同管理做成左侧独立模块
- 不引入客户数据范围权限
- 不把风险评估、负债登记改成客户汇总视图

## 信息架构

### 左侧菜单

新的左侧结构按以下方式调整：

- `客户管理`
  - `客户管理`
  - `风险评估`
  - `负债登记`
- `经营管理`
  - `商机管理`

现有 `系统管理`、`经营台账`、`财务管理` 结构保持不变。

### 页面关系

- `客户管理`
  - 主列表
  - 从列表进入客户档案页（新增/编辑）
- `风险评估`
  - 独立列表页
  - 可直接新增/编辑单条风险评估记录
  - 新增/编辑结果须在客户档案页同步可见
- `负债登记`
  - 独立列表页
  - 可直接新增/编辑单条负债记录
  - 新增/编辑结果须在客户档案页同步可见
- `合同管理`
  - 仅作为客户档案页内的子区块，不提供左侧入口

## 页面设计

### 1. 客户管理列表

列表字段需按最新口径对齐：

- 客户ID
- 客户名称
- 联系电话
- 公司名称
- 统一社会信用代码
- 所属行业
- 经营地址
- 测试日期
- 测试额度
- 流量
- 综合评分
- 龙信商
- 操作

说明：

- 测试日期、测试额度、流量、综合评分、龙信商取该客户最新一条风险评估记录的摘要
- 若客户暂无风险评估记录，对应字段显示 `-`
- 列表上的“新增”进入完整客户档案页
- 导入档案、导出按钮置灰

### 2. 客户档案页

客户档案页由四个区块组成：

#### 基本信息

字段如下：

- 姓名
- 性别
- 客户资料原件
- 身份证号
- 出生年月
- 年龄（按出生年月自动计算）
- 税务登记是否正常
- 企业名称
- 统一社会信用代码
- 成立日期
- 手机号码
- 经营地址
- 行业
- 开户行
- 银行开户账号
- 推荐人
- 推荐人返点
- 服务费
- 客户状态

约定：

- 税务登记是否正常：新增主表布尔字段
- 客户资料原件：支持多个附件
- 提交时与客户主档一起保存

#### 客户风险综合评估

字段如下：

- 测试日期
- 测试额度
- 流量
- 综合评分
- 龙信商

约定：

- 综合评分显示数值，新增时手动输入
- 用户填什么，列表就显示什么
- 龙信商使用独立字段，不和“评分通过/不通过”混用

#### 负债情况登记

字段如下：

- 负债类型
- 负债总额
- 已垫还
- 剩余需还额度
- 每期金额
- 还款日
- 备注

口径约定：

- `总需还 = 负债总额`
- `剩余需还额度 = 负债总额 - 已垫还`
- 现有客户负债链路中的 `repaidAmount` 语义重定义为 `已垫还`

#### 合同管理

字段如下：

- 客户名称
- 公司名称
- 统一社会信用代码
- 合同编号
- 合同名称
- 合同附件
- 备注

约定：

- 合同附件支持多个附件
- 客户名称、公司名称、统一社会信用代码允许手填
- 本轮不做左侧独立模块

## 数据模型

### 客户主表

现有 `cust_customer` 继续作为客户主表。

新增字段：

- `tax_registration_normal`：税务登记是否正常

继续使用现有字段：

- `customer_name / gender / id_card / birthday / age / mobile`
- `company_name / credit_code / established_date`
- `industry / business_address`
- `bank_name / bank_account`
- `recommender_name / recommender_rate`
- `service_fee / biz_status / remark`

### 风险评估子表

现有 `cust_customer_score` 继续使用。

字段映射：

- `test_date`
- `test_limit`
- `traffic_value`
- `composite_score`
- `third_party_score`
- `remark`

其中：

- `traffic_value` 对应“流量”
- `third_party_score` 对应“龙信商”

### 负债登记子表

现有 `cust_customer_debt` 继续使用，但语义调整如下：

- `debt_amount`：负债总额
- `repaid_amount`：已垫还
- `pending_amount`：剩余需还额度
- `installment_amount`：每期金额
- `repayment_day`：还款日

页面上 `总需还` 不单独落字段，直接由 `debt_amount` 映射展示。

### 合同子表

现有 `cust_customer_contract` 继续使用，但需从“单附件”扩展成“多附件”能力。

实现策略：

- 合同主记录继续保留在 `cust_customer_contract`
- 多附件通过 `sys_file` 关联到合同记录
- 不继续依赖单一 `contract_file_id` 作为唯一附件入口

### 客户资料原件

客户资料原件不落到客户主表字段中，改为：

- 通过 `sys_file` 以 `biz_type + biz_id` 方式关联客户
- 支持多个附件

## 后端接口设计

### 客户档案提交

新增或重构客户保存接口，使其支持整页提交：

- 创建客户档案：一次提交客户主信息 + 风险评估记录 + 负债记录 + 合同记录 + 客户附件引用
- 编辑客户档案：一次提交整页更新

事务要求：

- 客户主表和三个子表在同一事务中保存
- 任一子表保存失败，整页回滚

### 风险评估接口

补成真实 CRUD：

- 客户下风险评估列表
- 客户下新增/编辑/删除
- 独立页面分页查询

### 负债登记接口

补成真实 CRUD：

- 客户下负债列表
- 客户下新增/编辑/删除
- 独立页面分页查询

### 合同接口

补成真实 CRUD：

- 客户下合同列表
- 客户下新增/编辑/删除
- 支持多附件引用

## 前端实现策略

### 客户管理页

- 保留现有列表页入口
- 删除简化 drawer 表单
- 改为打开完整档案页或大尺寸档案抽屉
- 统一通过一个 `CustomerArchivePayload` 向后端提交

### 风险评估 / 负债登记页

- 页面独立
- 通过客户筛选查询记录表
- 新增/编辑表单字段和客户档案页内的子表单保持一致

### 菜单与权限

- 调整 `menu.ts`、`navigation.ts`、`AppLayout.vue`、`router/index.ts`
- 支持客户管理一级分组下的二级展开菜单
- 原有普通用户模块权限机制继续生效
- `visibleModuleKeys` 需要扩展支持：
  - `customer-management`
  - `customer-risk`
  - `customer-debt`
  - `opportunities`

## 验收标准

### 页面与交互

- 左侧菜单按新结构显示
- 客户档案页能完成一次整页提交
- 风险评估页面支持独立列表、新增、编辑、客户筛选
- 负债登记页面支持独立列表、新增、编辑、客户筛选
- 商机管理已迁移到 `经营管理`
- 导入导出按钮置灰，不触发真实上传/下载任务

### 数据与联动

- 新增客户档案后，客户主表和子表数据都能落库
- 从风险评估/负债登记独立页面新增或编辑后，客户档案页同步能看到
- 客户列表能正确展示最近一条风险评估摘要
- 负债登记列表的 `总需还 / 已垫还 / 剩余需还额度` 口径正确

### 工程质量

- 前端 `npm run build` 通过
- 前端新增单测通过
- 后端新增单测通过
- 本地联调链路不回退

## 风险

- 菜单二级结构需要改动共享布局文件，必须一次收稳，避免影响其他模块
- 文件上传接口当前仍是占位返回，多附件本轮先以“文件元数据引用”方式打通，不做真实存储实现
- 客户档案页一次提交会牵涉多表事务，若后端 DTO 设计不清晰，容易导致更新语义混乱
