# 华彩系统工作台 UI 重构设计

## 目标

在现有一期工程骨架之上，为前端后台建立一套可以直接继续开发的统一 UI 语言，优先覆盖：

- 主框架壳层
- 工作台首页
- 列表页骨架
- 抽屉表单
- 确认弹窗
- 常用状态标签、按钮、筛选区、分页和统计卡

本轮设计的目标不是做一套展示型视觉稿，而是先把后台 UI 的基础语言定准，让后续客户、借贷、财务模块可以在同一套规则下持续扩展。

## 已确认的设计结论

本次设计结论基于 2026-04-04 的协作确认：

- 产品气质：`专业作业台`
- 信息密度：`高密度但分层`
- 首页结构：`数据概览 + 待办 + 快捷入口的混合型`
- 列表骨架：`顶部统计条 + 折叠筛选 + 表格`
- 弹层策略：`新增/编辑用抽屉，确认/轻表单用弹窗`
- 组件覆盖：`统计卡、筛选区、表格、分页、状态标签、按钮、Tabs、批量操作条、空状态、抽屉、弹窗`
- 响应式目标：`桌面优先，同时兼顾笔记本和窄屏`
- 视觉方向：`冷静蓝灰`
- 最终选定配色版本：`A1 钴蓝企业版`

## 当前问题

现有前端骨架已经具备业务壳层，但 UI 语言仍停留在“原型承接阶段”，主要问题包括：

- 暖米色与棕橙主色更偏早期商务骨架，不够贴合专业作业台
- 页面标题、按钮、卡片圆角整体偏大，视觉颗粒度较粗
- 工作台、列表、弹窗之间尚未形成统一的结构语言
- 现有页面更像“占位页”，还不足以直接作为后续模块扩展模板

## 设计原则

### 1. 先服务作业效率，再做视觉精致

界面首先要服务业务人员的筛选、查询、查看、录入和批量操作。精致感来自比例、层次和留白控制，而不是夸张装饰。

### 2. 高密度但分层

主页面允许承载较多信息，但必须通过以下方式降低压迫感：

- 概览信息与操作信息分层
- 复杂筛选默认折叠
- 重录入场景放入抽屉
- 轻确认场景放入弹窗
- 次要信息进入侧栏或二级区块

### 3. 列表是后台核心，不让图表喧宾夺主

工作台首页可以有统计与图表，但主列表仍然是最重要的信息承载区域。图表、待办和快捷入口应服务主流程，而不是夺走主要注意力。

### 4. 精致感来自收敛

本轮设计不采用厚重卡片、超大按钮、强渐变和过度玻璃感。重点通过更细的边框、更轻的阴影、更小一档的字体和更克制的圆角来建立成熟感。

## 适用角色与核心场景

主要使用角色：

- 业务经办
- 客户经理
- 财务录入人员
- 审批与复核人员
- 管理视角使用者

核心场景：

- 在工作台查看今日任务、关键指标、异常提醒
- 在列表页通过条件筛选并执行批量操作
- 通过抽屉完成新增、编辑、补录
- 通过弹窗完成删除、提交、确认类动作
- 在统一壳层下切换客户、借贷、财务等模块

## 视觉系统

### 配色

选定 `A1 钴蓝企业版` 作为本轮主基线。

主色体系：

- 主色 `Primary`：`#1E40AF`
- 主色 hover `Primary Hover`：`#1D4ED8`
- 主色高亮 `Primary Accent`：`#3B82F6`
- 主色弱化背景 `Primary Soft`：`rgba(59, 130, 246, 0.12)`
- 页面背景 `Background`：`#F8FAFC`
- 主面板背景 `Surface`：`#FFFFFF`
- 次级面板背景 `Surface Soft`：`#F6F9FF`
- 主文字 `Text`：`#132238`
- 次文字 `Text Soft`：`#617186`
- 边框 `Border`：`rgba(148, 163, 184, 0.24)`
- 警告色 `Warning`：`#D97706`
- 危险色 `Danger`：`#D9485F`
- 成功色 `Success`：`#0F766E`

使用规则：

- 蓝色只负责激活态、主按钮、关键数字、关键标签、图表主序列
- 黄色与红色只用于预警与异常，不参与大面积铺色
- 大面积背景保持低饱和中性，避免整个后台变成“满屏蓝色”

### 字体与层级

本轮不引入额外在线字体，先沿用项目现有中文系统字体策略，通过字号、字重和行高建立精致感。

字号体系：

- 页面主标题：`26px`
- 区块标题：`16px`
- 首页一级 KPI 数字：`26px`
- 普通卡片数字：`24px`
- 正文说明：`14px`
- 标签、辅助说明、表头次信息：`12px`
- 按钮文字：`12px`
- 表格正文：`13px`

字重体系：

- 页面主标题：`600`
- 区块标题：`600`
- 普通正文：`400`
- 重点数字和关键状态：`600`

原则：

- 页面标题从“展示型大标题”收成“产品型标题”
- 尽量避免 30px 以上的大号标题长期占据后台页面
- 表格、筛选、标签、按钮统一收小一档，建立专业感

### 圆角、边框与阴影

圆角体系：

- 大面板：`20px`
- 普通卡片：`16px`
- 筛选区、次级区块：`12px`
- 按钮与输入框：`10px`
- 状态标签：`999px`

阴影体系：

- 页面主卡片：轻中层阴影
- 次级卡片：极轻阴影或仅边框
- 抽屉与弹窗：允许更明显的浮层阴影

原则：

- 阴影只用于区分层级，不作为主要装饰
- 不使用厚重、发灰、模糊范围过大的旧式后台阴影

## 布局设计

### 主框架壳层

主框架由左侧导航、顶部页面头部、主内容区组成。

壳层设计要求：

- 左侧导航稳定，不追求过宽
- 当前激活菜单使用钴蓝弱底和蓝色文字表现
- 顶部页面头部保留当前页标题和少量全局动作
- 用户区与页面动作区视觉上不应争抢主标题

壳层目标文件：

- `frontend/src/layout/AppLayout.vue`
- `frontend/src/style.css`

### 工作台首页

首页采用“三层结构”：

1. 顶部：页面标题、当前视角说明、主操作按钮
2. 中部：四个以内的关键统计卡
3. 主体：左侧主列表区 + 右侧辅助信息区

首页主体规则：

- 左侧主区承载借贷或经营列表，作为真正工作入口
- 右侧辅助区承载回款趋势、今日待办、快捷入口
- 图表保留，但不能压过列表
- 快捷入口变轻，不做大面积显眼按钮墙

目标文件：

- `frontend/src/views/dashboard/WorkbenchView.vue`

### 列表页骨架

列表页统一采用：

- 顶部统计条
- 折叠筛选区
- 状态标签与批量操作区
- 主数据表格
- 分页区

列表页原则：

- 默认展示核心筛选项
- 复杂筛选通过“展开筛选”进入更多字段
- 批量操作只在有选中项时提升存在感
- 列设置、导出、刷新作为次级操作，不与主按钮抢层级
- 表格是页面视觉中心

适用页面：

- 客户管理
- 借贷管理
- 还款明细
- 收入管理
- 支出管理
- 系统用户、角色等后台台账页

### 抽屉与弹窗

规则明确区分：

- 抽屉：用于新增、编辑、补录、查看较完整信息
- 弹窗：用于删除确认、提交确认、状态切换确认、轻量表单

抽屉要求：

- 视觉层次明显，但不做过度装饰
- 标题、辅助说明、表单区和操作区分层清楚
- 表单错误需要明确出现在字段附近

弹窗要求：

- 文案短、动作清楚
- 强调确认成本，而不是承载复杂输入

## 组件规则

### 按钮

按钮以 `34px` 到 `36px` 高度为主，不再使用偏高的大按钮。

规则：

- 主按钮用钴蓝实底
- 次按钮采用白底浅边框
- 文字按钮或轻操作按钮降低存在感
- 页面内同一区域不出现多个竞争性的主按钮

### 统计卡

统计卡强调“数据可读”，不强调装饰。

规则：

- 每页统计卡控制在 3 到 4 个
- 只有最关键的卡片可以使用主色弱底
- 数字较大，但不使用夸张大号字体
- 次级说明一律小号、低饱和

### 状态标签

状态标签用于表达状态，不可只靠颜色传达信息。

规则：

- 使用文字 + 色彩
- 蓝色用于信息和选中
- 黄色用于待处理与风险提醒
- 红色用于异常和危险
- 绿色用于完成、正常、恢复

### 筛选区

筛选区是列表效率的关键部分，设计上要轻但清楚。

规则：

- 使用浅底与轻边框容器
- 默认展示核心字段
- 次级字段通过展开区显示
- Tabs、状态筛选和搜索框层级清楚，不互相挤压

### 表格

表格是本轮设计中的一级组件。

规则：

- 表头弱底处理，降低重量
- 行高适中，支持高密度但不拥挤
- 表格行 hover 和选中状态清楚
- 批量选择、状态标签、行内操作区需要统一语言
- 窄屏下允许横向滚动，不强行压缩列宽

## 交互流设计

### 工作台到业务处理

推荐交互流：

1. 用户进入工作台
2. 查看关键指标与异常提醒
3. 进入主列表区筛选目标数据
4. 通过行内操作或主按钮进入抽屉
5. 在抽屉中完成新增、编辑或补录
6. 对危险或不可逆动作，通过弹窗确认

### 列表到抽屉

推荐交互流：

- 点击“新增”打开抽屉
- 点击表格行或“编辑”打开抽屉
- 抽屉保存成功后返回列表并刷新局部数据
- 复杂编辑不跳转到新页面，优先保持在当前上下文中完成

## 状态与错误处理

本轮实现必须覆盖以下状态：

- 加载态
- 空状态
- 接口报错状态
- 表单校验错误态
- 权限不足或操作不可用状态

设计要求：

- 表格加载时提供骨架或明确 loading 态
- 空状态需要有明确说明和下一步引导
- 表单错误信息必须落在字段附近，不能只靠颜色提示
- 状态标签和错误提示不能只通过颜色区分

## 响应式策略

本轮不以手机后台为目标，重点覆盖：

- `1440px` 桌面
- `1280px` 笔记本
- `1024px` 窄屏桌面

规则：

- `1280px` 以下时，右侧辅助区可以下移
- `1024px` 以下时，表格允许横向滚动
- 侧边导航在窄屏下可以收缩或切换为顶部结构
- 保证主要操作始终可见，不因适配牺牲核心流程

## 可访问性与可用性要求

基于本轮使用的 `ui-ux-pro-max` 规则，落地时必须满足：

- 普通文字对比度至少达到 `4.5:1`
- 图标按钮提供可识别名称
- 键盘导航顺序与视觉顺序一致
- 交互控件保持明确 focus 状态
- 状态信息不能只靠颜色表达
- 确认类操作文案要明确动作后果

## 第一阶段实现范围

第一阶段只做“建立 UI 基线”，不一次性重做全部业务页面。

第一阶段实现内容：

- 统一全局设计 token 与页面基础样式
- 重构主框架壳层
- 重构工作台首页
- 为后续列表页沉淀统一骨架和样式规则

第一阶段必须触达文件：

- `frontend/src/style.css`
- `frontend/src/layout/AppLayout.vue`
- `frontend/src/views/dashboard/WorkbenchView.vue`

第一阶段可新增的共享组件：

- 更通用的统计卡
- 页面区块头组件
- 状态标签样式
- 工作台快捷入口卡片

## 暂不纳入本轮范围

- 全量业务模块页面逐个重构
- 真实图表库接入
- 自定义品牌字体接入
- 复杂动画系统
- 手机端完整后台适配

## 测试与验收

实现后至少验证：

- `npm run build` 可以通过
- 工作台首页在 `1440 / 1280 / 1024` 下视觉稳定
- 列表区、侧栏、抽屉和弹窗层级关系清楚
- 颜色、字号、按钮高度已明显收敛
- 主色切换为 `A1 钴蓝企业版` 后，不再出现“过灰”的主观问题
- 键盘 focus 和对比度不出现明显可用性问题

## 风险与说明

- 当前目录不是 git 仓库，本轮无法提交设计文档 commit。
- 当前工程内只有少量前端页面，列表页与抽屉页还需要在后续实现中逐步沉淀为真实组件模板。
- 如果后续引入图表库或更复杂组件，仍需遵守本设计中的密度与层级规则，避免重新变回“展示型后台”。
