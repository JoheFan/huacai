# 华彩系统用户直配模块权限设计

## 背景

当前华彩系统已经具备登录认证，但还没有真正落地“登录后看什么”的权限控制：

- 前端左侧菜单由静态配置直接渲染
- 路由守卫只判断是否登录，不判断模块权限
- 后端接口只要求 `authenticated()`，没有模块级访问控制
- 普通员工与管理员登录后看到的入口和接口能力基本一致

本轮目标不是做完整 RBAC 平台，而是在一期范围内补一层**最小可用的模块级访问控制**。

## 目标

实现以下能力：

1. 新增/编辑用户时，管理员可以直接勾选该用户可见的业务模块。
2. 普通用户登录后，只能看到被授权的模块。
3. 普通用户手工访问未授权页面时，会被前端拦住。
4. 普通用户调用未授权模块对应接口时，会被后端拦住并返回 403。
5. 普通用户不进入经营工作台，登录后进入一个欢迎页。

## 范围

### 本轮要做

- 用户维度的模块可见配置
- 左侧菜单过滤
- 路由访问拦截
- 后端接口前缀拦截
- 普通用户欢迎页

### 本轮不做

- 按钮级权限
- 字段级权限
- 数据范围隔离
- 动态菜单中心
- 角色菜单、用户菜单双层覆盖
- 财务模块和系统管理模块对普通用户开放

## 模块边界

### 管理员

- `ADMIN` 用户默认拥有全部模块访问权限
- 管理员不需要在用户抽屉中手工配置模块
- 系统管理模块只对管理员开放：
  - 组织管理
  - 系统用户
  - 角色管理

### 普通用户

普通用户仅允许配置当前已启用业务模块：

- `customers` 对应 客户管理
- `opportunities` 对应 商机管理
- `loan-orders` 对应 借贷管理
- `repayments` 对应 还款明细

当前不开放：

- `dashboard`
- 所有 `system.*`
- 所有财务模块
- 所有占位模块

## 权限模型

采用“用户直接勾模块”的轻量模型，不引入完整动态菜单系统。

### 数据表

新增表：`sys_user_module`

字段建议：

- `id`
- `user_id`
- `module_key`
- `created_by`
- `created_at`

约束建议：

- `UNIQUE KEY uk_sys_user_module_user_module (user_id, module_key)`
- `INDEX idx_sys_user_module_user_id (user_id)`

### module_key 清单

后端与前端都维护同一份稳定 key：

- `customers`
- `opportunities`
- `loan-orders`
- `repayments`

## 前端设计

### 用户抽屉交互

位置：

- 在系统用户抽屉中，放在“账号状态”之后、“备注”之前
- 占整行，不做单行输入框

结构：

- 标题：`可见模块`
- 次说明：`未勾选的模块不会出现在左侧菜单，对应接口也不可访问`
- 分组：
  - 客户经营：客户管理、商机管理
  - 经营台账：借贷管理、还款明细
- 操作：
  - `全选`
  - `清空`
  - `恢复默认`
- 摘要：
  - `已选 N 个模块`

交互原则：

- 复选项整块可点击，不只点 checkbox
- 编辑用户时完整回填
- 非管理员用户至少选择 1 个模块
- 管理员账号显示只读说明：`管理员默认拥有全部模块权限`

### 左侧菜单

- 对现有静态菜单做过滤，不重做整体导航
- 非管理员仅显示：
  - 欢迎页
  - 被授权的业务模块
- 管理员维持当前全量菜单

### 路由守卫

- 已登录但访问未授权业务模块时：
  - 非管理员跳转到 `/welcome`
- 若用户完全没有业务模块：
  - 进入欢迎页空态

### 普通用户欢迎页

新增一个轻量欢迎页：

- 标题：欢迎回来
- 副标题：根据当前账号权限展示可访问模块
- 内容：
  - 当前可访问模块卡片列表
  - 每张卡片包含标题、说明、进入按钮
- 空态：
  - `当前账号暂未配置可访问模块，请联系管理员`

该页面不展示全局经营统计，避免误导为管理员视角。

## 后端设计

### 用户接口

在以下 DTO / VO 中增加 `visibleModuleKeys`：

- `UserCreateRequest`
- `UserUpdateRequest`
- `UserVO`
- `CurrentUserInfoVO`

行为：

- 创建用户时保存 `visibleModuleKeys`
- 编辑用户时覆盖更新 `visibleModuleKeys`
- 用户详情接口回填 `visibleModuleKeys`
- `auth/me` 返回 `visibleModuleKeys`

### AuthUser

在 `AuthUser` 中增加：

- `visibleModuleKeys`
- `admin` 或通过 roles 判断的便捷方法

### 模块注册表

后端维护一份静态模块注册表，至少包含：

- `moduleKey`
- `routePath`
- `apiPrefixes`

用于集中做接口判定，不在各个 controller 上分散写死。

### 接口拦截

新增一层模块访问校验器，按请求路径匹配模块：

- `/api/v1/customers` -> `customers`
- `/api/v1/opportunities` -> `opportunities`
- `/api/v1/loan-orders` -> `loan-orders`
- `/api/v1/repayments` -> `repayments`

规则：

- `ADMIN` 放行
- 非管理员必须命中自己的 `visibleModuleKeys`
- 未授权返回 403
- 不属于业务模块控制范围的接口不拦截：
  - `/api/v1/auth/**`
  - `/api/v1/health`
  - Swagger
  - 当前系统管理接口默认保留给管理员，建议同步做角色级限制

## 系统管理接口约束

本轮既然明确“系统管理只给管理员”，建议一起补最小角色限制：

- `/api/v1/system/orgs/**`
- `/api/v1/system/users/**`
- `/api/v1/system/roles/**`

这组接口只允许 `ADMIN` 访问。

这样能避免普通用户就算不显示菜单，仍然能直接调用系统管理接口。

## 默认策略

### 管理员

- 默认全部模块可见
- 不依赖 `sys_user_module`

### 普通用户

- 新建时至少勾选 1 个模块
- 若历史用户没有任何模块配置：
  - 登录成功
  - 进入欢迎页空态
  - 业务路由全部不可进入

## 非目标说明

这轮实现的是“模块访问控制”，不是“数据范围权限”。

例如：

- 给普通用户开了 `客户管理`
- 他仍会看到当前客户模块下的全部客户数据

后续如果需要“只能看自己客户 / 本组织客户”，应在下一轮单独做数据范围。

## 验收标准

1. 管理员在新增/编辑用户抽屉中可以直接配置普通用户可见模块。
2. 普通用户登录后看到欢迎页，不进入经营工作台。
3. 普通用户左侧菜单只显示被授权的业务模块。
4. 普通用户手工输入未授权业务路由时会被重定向到欢迎页。
5. 普通用户调用未授权模块接口时后端返回 403。
6. 管理员仍保持当前全量访问能力。
7. 前端构建通过，已有一期主链不回归。

## 风险

### 1. 静态菜单与静态模块注册表双维护

短期可接受，但需要保证前后端 `module_key` 一致。

### 2. 历史账号无模块配置

需要欢迎页空态兜底，否则会出现登录成功但无可访问入口的体验问题。

### 3. 还未做数据范围

用户虽然被限制模块入口，但在授权模块内仍是全量数据视角。

## 本轮建议

本轮就按以下顺序推进：

1. 落 `sys_user_module`
2. 后端保存/返回 `visibleModuleKeys`
3. 后端做模块接口拦截和系统管理 `ADMIN` 限制
4. 前端用户抽屉加模块配置区
5. 前端菜单、路由、欢迎页联动
6. 做真实登录与 403 验证
